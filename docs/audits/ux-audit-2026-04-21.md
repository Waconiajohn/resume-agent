# CareerIQ UX Audit

**Date:** 2026-04-22
**Scope:** Full-app UX audit through the eyes of a first-time user. All 13 product areas per the audit plan.
**Methodology:** Signed in as `jjschrup@yahoo.com` (test candidate account, has prior data). Drove a live Playwright browser against `localhost:5173` / `localhost:3001`, clicked through each surface, captured viewport + full-page screenshots as evidence, read console messages for silent failures. Surfaces that required a completed prerequisite (e.g., Interview Prep after scheduling an interview) are flagged where observation stopped. Components with no UI entry point (Exec Bio) were verified via code inspection and flagged as **code-inferred** where noted.

**Author:** Claude (Sonnet-class), observation-only pass. No code changes were made.

---

## Executive Summary

**Total issues found:** 34 (9 Blocker / 13 Major / 9 Minor / 3 Polish)
**Distinct product areas touched:** 12 of 13 (mobile out of scope)
**Top-level takeaway:** Approach C shipped the URL-scoped application workspace correctly, but the five tools inside it operate with near-zero context awareness of the parent application. The user has to re-enter company / role / JD for every tool despite having just entered them at application creation. Separately, the primary deliverable (the tailored resume) has no download button — confirming the owner's originating concern that testing has emphasized "does the code work" over "can a user accomplish their goal."

### Top 5 most impactful issues

1. **V3 resume has no download / export button.** (Blocker, Area 3.) The user completes the full pipeline and can "save to knowledge base," but cannot download a PDF/DOCX/TXT of the generated resume. The export libraries exist in `app/src/lib/export-pdf.ts` and `export-docx.ts` and are wired into v2's `ExportBar.tsx`, but v3 has no corresponding component mounted.
2. **Context does not pass through from the application to its tools.** (Blocker, Area 2.) After creating an application with role / company / JD, the resume intake, cover letter, and networking tools all render with those fields empty. User retypes. Only the thank-you note tool prefills company and role.
3. **"My Applications" sidebar button does not navigate.** (Blocker, Area 12.) Clicking it from the dashboard sets the button's `[active]` state but URL stays at `/workspace` and content does not change. Direct navigation to `/workspace/applications` works. A first-time user who can't guess the URL hits a dead end here.
4. **Sidebar disappears entirely inside the application workspace.** (Major, Area 2.) Once a user is on `/workspace/application/:id/:tool`, they have no global navigation. The only way back to the My Applications list is the browser back button. No breadcrumb either.
5. **Cross-application state leak in the resume "Pick up where you left off" banner.** (Major, Area 3.) Inside the Northwind Labs application workspace, the resume screen's top banner advertises a resume for "Director of Sales — Public Sector at RegionalCloud" (a different application). Clicking "Resume" would carry the user out of their current application context.

### Recommended sprint breakdown (detailed later in the doc)

- **Sprint A — Context pass-through + download wiring** (1 sprint, unblocks primary flows)
- **Sprint B — Application workspace nav & data consistency** (1 sprint)
- **Sprint C — Silent API errors + dead clicks** (half sprint)
- **Sprint D — Orphaned components + tool entry polish** (half sprint)
- **Sprint E — Auth / first-run / onboarding polish** (half sprint)
- **Sprint F — Mobile + accessibility audit** (placeholder; not audited here)

## Findings by Product Area

### 1. Authentication and first-run

#### User journey walkthrough

A first-time visitor lands on `/sales`, which presents a hero ("Your resume undersells you. We fix that."), ICP segmentation, a 4-step methodology, a FAQ, and two CTAs that both route to `/workspace`. There is no standalone sign-up form visible on the sales page — "Sign In" in the header and "Get started free" both go to the same gate. The auth gate is a centered card with email, password, Forgot password, Sign In, Continue with Google, and a "Don't have an account? Sign up" link at the bottom. After sign-in, a first-time user lands on `/workspace` with a "Complete your Career Assessment" hero card and three quick-action cards (Career Assessment / Resume work / Job board). The Help button in the top nav launches an 8-step onboarding tour dialog.

#### Friction points and gaps

- **"Sign In" and "Get started free" point to the same URL.** Severity: Minor. Cause: probably intentional (auth gate handles both), but a net-new user may not immediately realize the gate also supports sign-up. Fix: Small — either disambiguate the CTAs or make the sign-up link more prominent on the gate.
- **Sales hero is functional but the marketing page is very tall on a 1440×900 viewport.** Severity: Polish. Cause: copy-heavy. Fix: Small — consider viewport-height hero.
- **Post-login dashboard state assumes "you haven't completed Career Assessment yet."** Severity: Minor. Cause: "Complete your Career Assessment" dominates the page even for users who have already done it (the test user has a master resume but the dashboard still prompts to "Complete your Career Assessment"). Fix: Medium — condition the hero on whether a positioning profile already exists.

#### Missing features a user would expect

- No email-verification flow observed.
- No "Terms of Service accepted" checkbox on the gate (links exist in the sales footer but not on the gate itself).
- No session expiry messaging — if a user comes back tomorrow with an expired token, the experience wasn't tested.

#### What's working well

- Google OAuth + password + forgot-password all on one clean gate.
- The onboarding tour exists, is 8 steps, and can be re-triggered from the Help icon. Discoverable.

### 2. Application-scoped workspace (Approach C)

#### User journey walkthrough

The user navigates to `/workspace/applications` (directly, since the sidebar link is broken — see below). The empty state reads "No applications yet — Start a new application to group its resume, cover letter, and interview prep in one place" with a New Application button. Clicking New Application opens an inline form: Role title, Company, Job posting URL (optional), Job description (optional). Submitting creates the application and routes to `/workspace/application/:id/resume`. The screen shows an Application header card with tool tabs — resume (active) / cover letter / thank you note / networking / interview prep — and the resume intake form below. Clicking tool tabs updates the URL segment correctly and unmounts/remounts the tool subtree (the state-reset fix).

#### Friction points and gaps

- **BLOCKER — Sidebar "My Applications" link does not navigate.** Clicking it from `/workspace` appears to fire (the button shows an `[active]` state) but the URL stays at `/workspace` and the content does not change. Direct navigation to `/workspace/applications` works. Cause: `WorkspaceLayout.tsx` passes `onNavigateRoute={(route) => navigate(route)}` to the Sidebar, so the handler fires — something further up is blocking or the navigate is being overridden. Needs debug. Fix: Small (hours) to root-cause.
- **BLOCKER — No global navigation inside application workspace.** The `/workspace/application/:id/:tool` route does NOT wrap in `<WorkspaceLayout>` (by design — `App.tsx` L747-756). So the sidebar does not render. The user has to use the browser back button to return to the My Applications list, or use the tool tabs to navigate within the current application. No breadcrumb either. Fix: Medium — add a lightweight back-to-applications link in the Application header, or wrap the route in WorkspaceLayout with conditional sidebar, or add a breadcrumb.
- **BLOCKER — Context does not pass through to tools.** After creating the Northwind Labs application with a JD, the resume intake shows the JD field empty, the company field empty, and the job title field empty. Same gap on cover letter (company blank) and networking (target company blank). Thank-you note is the exception — it prefills company + role. Cause: the individual tool components (V3IntakeForm, CoverLetterScreen form, NetworkingHubRoom Outreach Generator) never read the `applicationId` prop to fetch the parent application. Fix: Medium — for each tool screen, read the application record from props or fetch by ID, and prefill relevant fields.
- **MAJOR — No "switch application" affordance inside workspace.** The header shows "Application / Northwind Labs" statically. Clicking it does not open a switcher. If a user wants to hop to another application, they must browser-back to the list. Fix: Small-medium — add a dropdown or make the company name a link.
- **MAJOR — No archive / delete on the Applications list.** Empty state didn't let me test, and the list view wasn't populated during this audit — but from code inspection there's no delete button in `ApplicationsListScreen.tsx`. Fix: Medium — requires backend DELETE handler (already exists in job-applications.ts) and a confirm-delete UI.
- **MINOR — Application header has no "last edited" or "last run" timestamp.** User has no way to see which of their applications are fresh vs stale. Fix: Small.

#### Missing features

- No application search / filter.
- No stage pill in the workspace header (the dashboard has one but the workspace just shows "Stage: saved" as inline text).
- No way to change JD after creation (doesn't re-flow into tools anyway).

#### What's working well

- URL-scoped routing is clean and the remount-on-id-change works as designed.
- Tab strip is readable and the active tab is visually distinct.
- Creating an application and landing directly on the resume intake is a sensible default.

### 3. Resume tooling

#### User journey walkthrough

From inside the application, the resume tab shows "Resume v3 — Attribution-first resume tailoring" with a banner "Pick up where you left off — Your last tailored resume is ready to resume for [X] at [Y]" (two buttons: Start fresh / Resume). Below is the "Tailor your resume" form: radio for Resume Source (Use my knowledge base vs. Upload a different resume), Job Description (fetch from URL OR drop zone OR paste), Job Title (optional), Company (optional), and a disabled Generate Tailored Resume button. The user fills the JD, clicks generate, the pipeline runs through extract → classify → strategize → write → verify, and displays the final resume via V3ResumeView. Below the resume is V3PromotePanel ("Save this run as your knowledge base" or "Save defaults to knowledge base").

#### Friction points and gaps

- **BLOCKER — No download / export button in V3.** (Owner-known gap.) After the pipeline completes, the user can view the resume, edit bullets, and "promote" selections to their knowledge base — but there is no PDF / DOCX / TXT export. The export libraries exist (`app/src/lib/export-pdf.ts`, `export-docx.ts`) and v2 has `ExportBar.tsx` wired; v3 never got the wire-up. Fix: Small-medium — build a V3→FinalResume adapter and mount ExportBar in V3PipelineScreen.
- **BLOCKER — "Pick up where you left off" shows a resume from a DIFFERENT application.** Inside the Northwind Labs application workspace, the banner advertised "Director of Sales — Public Sector at RegionalCloud." Clicking "Resume" presumably lifts the user out of the current app context. Cause: the "last session" lookup is user-scoped, not application-scoped. Fix: Medium — scope the banner to the current application or suppress it when inside an application workspace.
- **MAJOR — "Promote to knowledge base" is easy to mistake for "save/export."** The primary post-generation CTA is "Save defaults to knowledge base." A user looking for a download button may click this, get a confirmation ("Saved as knowledge base v1"), and still have no file. The label is technically correct but misleading for the common task. Fix: Small — relabel or add an explicit "Export" affordance before the promote card.
- **MAJOR — Data inconsistency between Resume Builder landing and application workspace.** Standalone `/workspace?room=resume` shows "Master Resume: MISSING" and "No default master resume yet" — but inside an application the resume intake's radio button says "Use my knowledge base — v2, last updated today · 3 positions · 2 evidence items." The same account, same data, contradictory representations. Fix: Medium — unify the data source or fix the Resume Builder landing to read the actual knowledge-base summary.
- **MINOR — Job title and company are marked (optional) despite being entered at application creation.** The user already gave us these values, but the pipeline still marks them optional and leaves them blank. Should be prefilled from the parent application. Fix: Small.
- **MINOR — No versioning / iteration history on a given application.** If the user generates twice, there's no surfaced way to compare the two runs or revert to the first. Fix: Medium.

#### Missing features

- **Export to PDF / DOCX / TXT** — the headline gap.
- **Copy to clipboard** — for pasting into application portals.
- **Print stylesheet** — the resume view is laid out in the app chrome; no CSS print rules.
- **Shareable link** — e.g., a view-only URL to share with a recruiter.

#### What's working well

- The intake form is well laid out; the JD drop zone supports `.txt .docx .pdf .html .htm` or URL fetch.
- The "Use my knowledge base" radio with version metadata is good transparency.
- The banner is the ONLY clear regression — the form below it is clean.

## Cross-cutting observations

### Pattern 1: Context pass-through is systematically broken.
- Application creation captures role / company / JD.
- Resume intake, cover letter, and networking ALL leave those fields blank on entry.
- Only thank-you note prefills.
- Root cause: individual tool components never read the parent application's data even though `applicationId` is threaded through.
- Touches: Areas 2, 3, 4, 6.

### Pattern 2: The application workspace has no global navigation.
- `/workspace/application/:id/:tool` doesn't wrap in `WorkspaceLayout`, so no sidebar, no breadcrumb.
- User relies on browser back and tool tabs.
- Touches: Areas 2, 12.

### Pattern 3: Two parallel "applications" tables leak through.
- `job_applications` (new, canonical) vs. `application_pipeline` (legacy kanban).
- Networking's Rule of Four reads from the legacy table and can't see job_applications rows.
- Job Search's Daily Ops hits `/api/applications/due-actions` which doesn't exist under the new route.
- Touches: Areas 6, 10. Phase 3 cleanup addresses this.

### Pattern 4: Silent API failures are widespread.
- 4+ distinct 404 / 429 errors observed on routine page loads.
- None surface to the user.
- Touches: Areas 5, 10, 13.

### Pattern 5: Primary outputs have no export path.
- V3 resume: no download.
- Cover letter: not verified but suspected.
- Thank-you note: not verified.
- Resume-v2 has ExportBar; none of the newer tools inherit.
- Touches: Area 3; likely Areas 4, 5.

### Pattern 6: Dead clicks.
- "Step 1 Prep" in Interview Prep.
- "Pipeline" tab in Job Search.
- "My Applications" in sidebar.
- Suggests general lack of click-through QA.
- Touches: Areas 2, 7, 10.

### Pattern 7: Orphaned components.
- Exec Bio: complete component, tests, route — but no import, no sidebar, no route.
- Suggests the codebase has drifted faster than the UI shell.
- Touches: Area 9.

### Pattern 8: Opaque labels / acronyms.
- "NI" in networking.
- "Save to knowledge base" meaning "promote bullets."
- "Rule of Four" is explained inline, but not on the first visit.
- Touches: Areas 3, 6.

### Accessibility observations (brief)

- Skip-to-main-content link on every page ✓.
- Radio groups, form labels, buttons have accessible names (verified via Playwright snapshot which surfaces the accessibility tree).
- Aria-expanded and role=menu present on user menu.
- Color contrast on `[var(--text-soft)]` / `[var(--text-muted)]` against the dark background not measured but looks acceptable.
- Not audited: keyboard-only navigation, screen reader flow, focus trap on modal dialogs (the onboarding tour dialog trapped focus correctly on Escape).

## Proposed sprint breakdown

### Sprint A — Context pass-through + download wiring (HIGHEST PRIORITY, 1 sprint)
**Why first:** These are the two most user-visible gaps. They affect every tool in the app and cause users to doubt that the application workspace actually groups their work.
**Closes:**
