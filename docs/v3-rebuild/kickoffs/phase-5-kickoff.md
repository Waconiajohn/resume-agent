# Phase 5 — Shadow Deploy, Cutover, and v2 Deletion

Paste this into Claude Code after Phase 4 is complete and the full v3 pipeline has been approved. This phase spans calendar time (shadow deployment needs real traffic) and cannot be compressed into a single autonomous sprint.

---

You are starting Phase 5 of the CareerIQ v3 rebuild. Re-read `CLAUDE.md` and doc 02 (migration plan) before starting. Phase 4's pipeline is approved and tested. Now we deploy it.

This phase is different from Phases 1-4. Most of the work is observation, not construction. The human is more involved because we're now making changes that affect real users.

## Goal of this phase

Shadow deploy v3 alongside v2. Compare real-traffic output. When v3 proves reliable, cut over. Then delete v2.

## Step 1 — Shadow deployment

Wire v3 to run in parallel with v2 on every real user request. Users see v2 output unchanged. v3 runs silently, logs its full output, and stores both v2 and v3 results for comparison.

Implementation:
- Add a shadow runner that fires v3 whenever v2 fires
- Store results in a comparison table (Supabase or flat log files — use your judgment based on scale)
- Do not block the user's response on v3 completing
- Do not surface v3 errors to the user — log them for analysis
- Sample rate: 100% of requests for the first week, to maximize signal

Create a dashboard or report view at `docs/v3-rebuild/reports/shadow-comparison.md` (regenerated by a script) that shows:
- Total shadow runs
- v3 completion rate
- v3 vs v2 output diff summary (structural: same number of positions? similar summary length? etc.)
- Specific examples where v3 and v2 disagree materially

Run shadow for a minimum of 3 days of real traffic, ideally longer if traffic volume is low.

## Step 2 — Shadow analysis

At the end of the shadow window, go through every case where v3 and v2 produced materially different output. For each:
- Is v3 better, worse, or different-but-equivalent?
- If worse, which stage is the weak link?
- Is this a prompt issue (fix in place) or an architectural issue (flag for human review)?

Iterate prompts based on real-traffic findings. Each prompt change triggers a fixture suite run to confirm no regression. Each prompt change gets a version bump per doc 03.

Do not cut over until shadow results show v3 ≥ v2 on 95%+ of real requests. This is the gate.

## Step 3 — Controlled rollout

When shadow metrics pass the gate, route 10% of traffic to v3 (real users see v3 output). Mechanism:
- Feature flag keyed on user ID hash or request ID
- Instant rollback: flip flag back to v2 without a deploy
- Monitor error rates, latency, user-reported issues closely

Run 10% for 24-48 hours. If issues appear, rollback and iterate. If clean, proceed.

Route 50% for another 24 hours. Clean? Route 100%.

## Step 4 — Full cutover

100% of traffic on v3. v2 code still exists in the repo as fallback. Keep v2 runnable for one week after full cutover. During this week:
- Monitor v3 error rates
- Collect user feedback
- Keep the rollback switch armed

If anything major surfaces, rollback to v2 and iterate. Do not proceed to Step 5 until the cutover week is clean.

## Step 5 — v2 deletion

Per doc 02, delete v2 code:

```
server/src/agents/resume-v2/candidate-intelligence/
server/src/agents/resume-v2/resume-writer/
server/src/agents/resume-v2/benchmark-candidate/
server/src/agents/resume-v2/gap-analysis/
server/src/agents/resume-v2/source-resume-outline.ts
server/src/agents/resume-v2/ (entire directory if empty after above deletions)
```

Delete the associated tests. Delete the guardrail functions. Delete `mergeCandidateExperienceWithSourceOutline`, all phantom filters, all coercion/salvage functions. Full list in doc 05's "Guardrail functions identified" section.

Keep:
- `server/src/lib/llm-provider.ts` (used by v3)
- Service account auth (used by v3)
- Route handlers (update to call v3)
- Shared types (update to match v3 output shape)

Rename `server/src/v3/` to `server/src/resume/`. Update all imports. The v3 tag was scaffolding; the production pipeline is just "the resume pipeline" now.

Run the fixture suite. Everything still passes. Commit.

## Step 6 — Post-deletion cleanup

The codebase is now significantly smaller. Audit what remains:
- Any imports to v2 paths that didn't get caught?
- Any config referencing v2 agents that's now dead?
- Any documentation referencing v2 architecture that needs updating?
- Any feature flags or routing logic that can be simplified now that there's only one pipeline?

Update the repo's main README to reflect the new architecture. Update CLAUDE.md at the repo root — the "rebuild" phase is over; future Claude Code sessions operate on the production pipeline, not a parallel v3.

## Constraints

- Every change during shadow analysis is a prompt change, not a code change (unless you find a bug in stage plumbing). The architecture is fixed; only prompts iterate.
- No shortcuts on the rollback week. If v3 looks clean after 3 days, still wait the full week before deleting v2. Deletion is irreversible; waiting is cheap.
- Do not delete v2 code piecemeal during rollout. Delete everything in one commit after the rollback window passes. This gives us a clean "before/after" diff and a single revert point if something catastrophic surfaces immediately after deletion.
- When renaming `v3` → `resume`, do it in a single commit with no other changes. This is the most searchable moment in the repo's history; keep it clean.

## Definition of done

- Shadow deploy completed, results documented
- 10% → 50% → 100% rollout completed without rollback
- One-week monitoring period post-cutover completed
- v2 code fully deleted from the repo
- Directory renamed from `v3` to `resume`
- Fixture suite still passes
- Success metrics from doc 02 measured and reported:
  - Pipeline p95 latency (target: under 60s)
  - Agent count (target: under 10)
  - Files in `resume-v2` directory (target: zero, directory deleted)
  - Guardrail functions (target: zero)
  - Manual patches per week post-launch (target: zero; measure over first two weeks)
  - User-reported output errors (target: under 5%)
- Phase report written to `docs/v3-rebuild/reports/phase-5-report.md`
