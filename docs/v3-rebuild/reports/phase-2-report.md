## 1. What I built

### Stage 1 (Extract) — real implementation

- `server/src/v3/extract/index.ts` — replaces the Phase 1 stub. Uses `mammoth.convertToMarkdown` for `.docx`, `pdf-parse` v2's `PDFParse` class for `.pdf`, pass-through for `.txt`/`.md`. Format is detected from filename extension AND from buffer magic bytes (`%PDF…`, `PK\x03\x04`), with buffer signature winning when they disagree. Emits warnings (not errors) for embedded images, U+FFFD replacement chars, and suspected multi-column layouts. Throws loudly on corrupted input — no silent fallback. Strips mammoth's markdown-escape sequences (`\-`, `\.`, `\+`) that would otherwise break plaintext PII scanners and confuse downstream prompts.
- `server/src/v3/types.ts` — `extract` input type accepted the Phase 1 minimal shape and now carries an optional `path` field so direct file extraction works from scripts (no semantic change — purely an additive convenience).

### Fixture corpus

- **20 raw fixtures** placed by the human into `server/test-fixtures/resumes/raw/` (all `.docx`). Directory is gitignored.
- **20 extracted text files** at `server/test-fixtures/resumes/extracted/<slug>.txt` produced by running Stage 1 against every raw. Gitignored until redaction.
- **20 metadata YAMLs** at `server/test-fixtures/resumes/meta/<slug>.yaml`. Each carries: `name`, `file`, `category`, `characteristics` map, human-readable `notes`. Categories match the Phase 2 kickoff diversity checklist. Gitignored.
- `server/test-fixtures/resumes/fixture-map.json` — generated by `scripts/extract-fixtures.mjs`, maps raw filename → slug. Gitignored alongside raw/.

### Slug conventions

- `server/src/v3/test-fixtures/slug.ts` — pure mechanical `slugifyFilename(name)`. Example: `"Ben Wedewer - Resume trimmed.docx"` → `"ben-wedewer-resume-trimmed"`. Wrapped with a `fixture-NN-` prefix by both the extract script and the runner, so both produce identical paths.

### Fixture runner enhancements

- `server/src/v3/test-fixtures/runner.ts` — Phase 2 upgrades:
  - Discovers fixtures via slug (`fixture-NN-<slug>`) and loads the matching `meta/<slug>.yaml`.
  - New flag: `--filter key=value` (repeatable) — matches `meta.category` OR `meta.characteristics[key]` exactly. Example: `npm run fixtures -- --filter category=executive` returns the 4 executive fixtures.
  - Existing `--only <slug>` now operates on slugs (previously raw basenames).
  - Builds a pipeline input from the actual file buffer (Phase 1 just passed a filename; Phase 2 reads the bytes).
  - Prints a fixed-width summary table: `slug | category | status | notes`.

### Scripts (one-shots for Phase 2 operations)

- `server/scripts/extract-fixtures.mjs` — runs Stage 1 on every raw file, writes `extracted/<slug>.txt`, emits a tabular summary. Invoked once during Phase 2 build-out; kept in the repo for reuse when raw/ changes.
- `server/scripts/pii-scan.mjs` — regex-scans extracted text for PII categories and writes a count-only, name-anonymous report.

### Reports

- `docs/v3-rebuild/reports/phase-2-pii-scan.md` — PII scan report. Describes **what kinds** of PII are present per fixture without reproducing any values. Commit-safe. See §3 below for the caveats.

### Dependencies

Three packages installed autonomously per OPERATING-MANUAL.md §"What you should do autonomously":

- `mammoth@^1.12.0` — DOCX → markdown.
- `pdf-parse@^2.4.5` + `@types/pdf-parse@^1.1.5` — PDF text extraction. Not exercised yet; no PDF fixtures.
- `js-yaml@^4.1.1` + `@types/js-yaml@^4.0.9` — metadata file loading. (gray-matter from Phase 1 is markdown+frontmatter only, unsuitable for standalone YAML.)

No new security advisories introduced. The three pre-existing moderate vulns (brace-expansion, follow-redirects, hono) from Phase 1 are unchanged.

### Extraction quality (sample)

I inspected 7 of 20 extracted outputs by eye (fixtures 01, 05, 06, 07, 14, 17, 19, 20):

- **Good quality (15 fixtures)**: clean plaintext with bullet markers (`-`, `•`), section headers (`__…__` or `#`), emphasis spans, and coherent top-to-bottom ordering. Mammoth's markdown output is a solid starting surface for the Phase 3 classify prompt.
- **Acceptable-with-quirks (3 fixtures)**:
  - **fixture-18 (Alexander)** has a narrated "open to work" block at the top that reads like a pseudo-position (May 2025–Present, "Steve Alexander Consulting"). This is a **critical test case** for Phase 3: it's the Tatiana career-gap pattern in resume-writer form — the classifier must not treat a "pursuing opportunities" block as a normal position entry.
  - **fixture-05 (Cockrill) and fixture-07 (Downs) and fixture-13 (Slagle)** each contain unfilled template placeholders (`__The Job Title here__`, `__(Summary of why you are the best fit for the position here) Example-__`) left over from the base template the user was working from. Phase 3 must detect and refuse-to-parse these as real content.
- **Low quality (2 fixtures)**:
  - **fixture-14 (LJ 2025)** — source DOCX is a multi-column layout with embedded PNG icons. mammoth inlines the base64 image bodies (hundreds of kilobytes of `data:image/png;base64,…` per image) in the middle of the extracted text, and the section ordering is scrambled (summary lands ~line 72, after the experience block). See §3c below.
  - **fixture-19 (Goodwin)** — source DOCX is a two-column layout with a date column on the left and section content below. Dates and company names end up in separate non-adjacent blocks in the extracted text. Will stress-test Phase 3's ability to associate dates to positions.

## 3. What is uncertain

These are places I made a call that the human should review.

### 3a. Fixture-20 is a job description, not a resume

The file `UA Account Manager, Wholesale - Mall.docx` in `raw/` is a job posting (Under Armour Account Manager, Wholesale – Mall role description), not a resume. It's useful — paired with any candidate, it becomes a Phase 4 (Strategize) input — but it must not be fed to classify. I flagged it in `meta/fixture-20-…yaml` with `category: NOT_A_RESUME` and `is_job_description: true`.

**Proposed options:**

1. Move the file out of `raw/` into a new `server/test-fixtures/job-descriptions/` directory.
2. Keep it in `raw/` but have Phase 3's classify entry-point check the meta `category` field and skip or route to a JD-specific path.
3. Rename the file prefix so the runner filters it out by convention (e.g., `jd-` instead of no prefix).

I did not move or rename the file — that feels outside v3 scope without your direction. Please tell me which of the three you prefer (or a fourth).

### 3b. Fixture-19's References section is third-party PII

Fixture-19 (Goodwin) ends with a References section listing 11 individual names + email addresses. These are real people who did not consent to being in a CareerIQ fixture corpus. The PII scan flags it (`refs ✓` on row 19, email count = 23).

**Proposed remediation:** before committing any extracted/ content, redact the References section manually in the extracted text file (leave the raw file alone). The classifier will then never see those names. Or delete the References section from the raw DOCX upstream. I did not do either — per OPERATING-MANUAL.md §"Do not redact unilaterally."

### 3c. Fixture-14 extraction quality is poor and unlikely to classify cleanly

mammoth inlined ~300KB of base64 image data mid-document and shuffled the section order (summary appears after experience). Options:

1. **Pre-process pass that strips data URIs** — purely mechanical, safe to add to Stage 1. Would reduce noise substantially. Not controversial.
2. **Replace the raw file with a non-templated version** (ask the candidate for a plain-format copy). Outside my scope.
3. **Accept as-is** and let Phase 3's classify prompt deal with noise — this is the "adversarial fixture" approach that keeps the quality gate honest.

I did **option 1's analogue** minimally: my `normalizePlaintext` doesn't strip `data:image/…base64,…` URIs. Adding that as a mechanical strip in Phase 2 would be a simple follow-up; I did not because stripping data URIs has no semantic consequence but does change the extracted text for every fixture, which would churn the snapshot baselines we haven't yet established.

Please advise: add the data-URI strip now (with a v3-test re-run), or defer to Phase 3 prompt engineering.

### 3d. Fixture categorization is my best-effort read, not gospel

I opened every extracted file, inferred category and characteristics, and wrote meta YAMLs. My judgment calls:

- I labeled fixture-05 (Casey Cockrill) gender `ambiguous`. If you know Casey is male or female, amend.
- I labeled fixture-08 (J.Vaughn / Jason) `has_template_placeholders: minor` because `__Supply Chain Professional__` at the top could be either a real title or a half-filled template. Call it.
- Fixture-18 (Alexander) I marked `has_career_gap: true` based on the "Actively pursuing new leadership roles" opener. Technically it's presented as a current consulting engagement; structurally it's a narrated gap. Judgment call.
- Fixture-03 (Dullack) I marked `has_career_gap: possible` because 1994 → 2008 is a 14-year silence between USMC and first listed civilian role. The resume might just elide early-career detail; genuine unknown.

The meta files are gitignored for now. If you want to confirm my reads, the files are at `server/test-fixtures/resumes/meta/` on disk.

### 3e. `loadMeta` casts to `FixtureMeta` without strict validation

My meta-file loader parses YAML via `js-yaml` and casts the result to `FixtureMeta`. If someone edits a meta YAML and breaks the shape (e.g., `characteristics: "not an object"`), the runner will happily accept the garbage and fail at the filter step with a cryptic error. Phase 3 or later should add a zod schema for `FixtureMeta`. I did not add it here because defining the schema before the shape stabilizes would front-load churn.

### 3f. pdf-parse is wired but not exercised

All 20 fixtures are DOCX. The PDF extraction path runs on zero real fixtures. The branch has a unit test (bogus PDF buffer → `pdf-parse failed` error thrown), but no positive-case coverage. If Phase 3 introduces a real PDF fixture, I'd expect `PDFParse.getText()` to work per the shape I probed (`{ text: string, total: number, pages: [] }`), but that's unverified against a real PDF.

### 3g. `runPipeline` trace lines 500+ logs lines per fixture run to stderr

When the runner iterates 20 fixtures, pino streams start/complete/fail logs to stderr. That's ~5 lines/fixture → ~100 lines at info level. Fine for CI, noisy for local runs. A `--quiet` flag or a log-level env var (`LOG_LEVEL=warn npm run fixtures`) would help. Not urgent; flagging.

### 3h. Stage 1's multi-column heuristic probably fires a false positive on fixture-19

My "more than half of non-empty lines under 20 chars → multi-column artifact" check does catch fixture-19's date column, which is good. But it may also trip on short-bullet-heavy resumes. Monitoring this warning's signal/noise in Phase 3 will tell us whether the threshold is right.
